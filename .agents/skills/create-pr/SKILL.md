# Create PR

## 使うタイミング

ユーザーが PR を作ってほしいと依頼したとき、ブランチを切って PR まで進めたいとき、`gh pr create` を使って GitHub に pull request を作成したいときに使う。

## このリポジトリで優先する流れ

1. 現在の変更内容を確認する。
2. まだ適切な作業ブランチでなければブランチを作る。
3. ブランチ名は既存運用に寄せる。
4. 必要なら remote に push する。
5. PR タイトルと本文を作る。

## ブランチ名ルール

- 既に適切なブランチにいるなら、そのまま使う。
- `main` にいる場合や、ブランチ名が作業内容を表していない場合は新しいブランチを作る。
- 基本は `feature/<topic>` を使う。

## 事前確認

PR 作成前に必ず次を確認する。

- `git status` で untracked を含む変更内容
- `git diff` で staged / unstaged の差分
- 現在ブランチが remote を追跡しているか
- `main` から分岐して以降の `git log` と差分

差分が空なら PR は作らない。

## PR タイトル

- 変更の目的がすぐ伝わる短いタイトルにする。
- コミット履歴に `fix(seo): ...` のような形式があるので、必要に応じて `fix(...)`, `feat(...)`, `chore(...)` を使ってよい。
- ただし無理に Conventional Commits に寄せず、変更内容が自然に伝わることを優先する。
- PR タイトルは「何をしたか」より「何のための変更か」が伝わるものにする。

## PR 本文

PR テンプレートが見当たらないため、本文は簡潔に次の形式を基本とする。

```markdown
## Summary
- 変更点の要約
- 変更点の要約

## Test plan
- [ ] 未実施
- [ ] ローカルで確認済み
```

- `Summary` は 1〜3 個の bullet に収める。
- `Test plan` は実際に行った確認内容に合わせて書く。
- テスト未実施なら未実施と明記する。

## 実行手順

1. 変更内容とブランチ状況を確認する。
2. 必要なら新しいブランチを作る。
3. remote に未 push なら `git push -u origin HEAD` を使う。
4. 作成後は PR URL を返す。

## コマンド方針

- GitHub 関連操作は `gh` を使う。
- push は必要なときだけ行う。
- 破壊的な git 操作はしない。
- ユーザーに頼まれていない commit や push を勝手に増やさない。
- PR 作成前に、PR に含まれるコミット全体を把握してから本文を書く。

## 例

- ブランチ名例: `feature/XXX`
- PR タイトル例: `feat(post): XXXに対応`
