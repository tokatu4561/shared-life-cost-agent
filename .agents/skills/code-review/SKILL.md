# Code Review

## 基本方針
- HEAD ブランチを本番（master ブランチ）にマージすることで起こり得る項目を実行順序に沿って調査してください。
- 対象は常に現在の HEAD ブランチが `master` にマージされた想定とする
- 調査にあたりテストコードの作成・実行、および `git fetch` の実行は行わない
- 発見事項は重大度順（高→低）で報告し、影響範囲や再現条件を明記する
- コミット内容で比較する
- コードは編集しないこと

## 実行順序

### エラーチェック
- HEAD を `master` にマージした結果、致命的なサーバーエラーが発生しないかを重点的に調査する
- 例外ハンドリング、null ガード、外部 API 例外時のフォールバックなどエラーパスを網羅的に確認する

### 影響範囲のチェック
- 新旧コードの差分が既存機能へ破壊的影響を与えないか確認する
- API 仕様／イベントフロー／データ構造の互換性と副作用を重点的に見直す

### クリティカルな問題チェック
- マージにより重大なバグ（データ欠損、認可漏れ、取引失敗など）が潜在しないかを確認する
- 重大なビジネスロジック変更点は要件と突き合わせて検証する

### 脆弱性チェック
- 変更が新たなクリティカル脆弱性（SQLi、XSS、認証回避、情報漏えいなど）を生まないか確認する
- 依存ライブラリの扱い、入力値検証、シークレット管理を重点的にレビューする
