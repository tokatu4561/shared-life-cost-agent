# 機能名

## 概要
この機能が何を実現するものかを簡潔に記載する。

## 背景・動機
なぜこの機能や仕様が必要なのか、どの業務課題や運用課題を解決したいのかを記載する。

## 目的
この機能が何を達成するために存在するのかを記載する。

例:
- 注文を確実に受け渡し完了までつなげる
- 在庫切れ商品の誤販売を防ぐ
- 店舗運用で必要な情報を適切なタイミングで扱えるようにする

## スコープ
本仕様書が対象とする範囲を記載する。

例:
- 対象となる業務フロー
- 記録対象となるイベントや状態

## 要件

### システムフロー

#### 基本フロー
1. 開始条件を記載する
2. 利用者またはシステムの操作を記載する
3. システムの処理を記載する
4. 完了条件を記載する

#### 例外フロー
1. 想定される例外ケースを記載する
2. エラー時や不正状態時の扱いを記載する
3. 再実行、差し戻し、拒否条件があれば記載する

### 機能要件

#### 要件カテゴリ1
- 機能として必要なことを記載する
- 業務上保証したいことを記載する

#### 要件カテゴリ2
- XXXX

### 非機能要件
- 必要があれば記載
- 認証、認可、監査、ログ、性能、冪等性などの要件を記載する
- 業務上重要な制約があれば記載する

## 受け入れ条件
- [ ] 基本フローが成立する
- [ ] 主要な例外フローが定義されている
- [ ] 業務上重要な状態や制約が明文化されている
- [ ] 関係システムの責務分担が明確である

## 補足
- コードから一意に分かる実装詳細は再記述しすぎない
- 仕様書には業務上の意味、制約、判断基準、責務分担を優先して書く
- コードの詳細(How)ではなく、Why(目的、コードに現れない判断基準)/What(機能・非機能要件、受け入れ条件など)を重視する
- 依存する機能があれば記載する
