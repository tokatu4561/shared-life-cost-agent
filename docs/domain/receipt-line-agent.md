# レシート読み取りLINE Agent

## 概要
LINEで送られたレシート画像を読み取り、同棲生活における共有支出の月末精算に必要な情報をGoogleスプレッドシートへ記録する。

本機能は単なる家計簿Botではなく、食費・日用品費などの立替精算を低負担で続けるための記録エージェントとして扱う。

## 背景・動機
同棲生活では、食費や日用品などの共有支出が日常的に発生する。月末に精算するためには、購入日、店舗、金額、レシート画像などの記録が必要になる。

手作業での転記は、記録漏れ、後回し、レシート紛失が起きやすい。そこで、普段使っているLINEへレシート画像を送るだけで、精算に必要な最低限の情報をスプレッドシートへ蓄積できる仕組みを作る。

## 目的
月末精算に必要な共有支出記録を、できるだけ低負担で残すことを目的とする。

具体的には、以下を実現する。

- レシート画像をLINEに送るだけで記録できる
- 食費・日用品を中心とした共有支出を記録できる
- 月末にスプレッドシートを見れば、精算対象の支出を確認できる
- レシート画像の参照先を残し、必要に応じて後から確認できる

## スコープ
MVPでは、1人がLINEへレシート画像を送る運用を前提とする。

### 対象
- 同棲生活の共有支出
- 主な記録対象は食費と日用品
- LINEで送信されたレシート画像
- Googleスプレッドシートへの記録
- レシート画像のS3保存

### 対象外
- 月次集計の自動化
- 支払者の管理
- 精算済み管理
- 精算対象外フラグ
- 分担率の管理
- 複数ユーザー対応
- 複数スプレッドシート振り分け
- 修正用の対話フロー
- 会計アプリ連携
- 管理画面

個人支出の厳密な除外はMVPでは自動判定せず、必要に応じてスプレッドシート上で手動修正する。

## 要件

### システムフロー

#### 基本フロー
1. 利用者がLINE Botへレシート画像を送信する。
2. Webhook LambdaがLINE署名を検証し、画像メッセージであることを確認する。
3. Webhook LambdaがLINE Messaging APIから画像バイナリを取得する。
4. Webhook Lambdaが画像をS3へ保存する。
5. Webhook Lambdaが受付メッセージをLINEへ返信し、SQSへ処理依頼を投入する。
6. Worker LambdaがSQSメッセージを受け取り、Agent Core Runtimeを呼び出す。
7. AgentがGoogle Cloud Vision APIでレシート画像からOCRテキストを抽出する。
8. AgentがLLMで抽出結果を補正し、記録用データへ正規化する。
9. 店舗名と合計金額が取得できた場合、Agentがレシート日付に対応する月別シートへGoogle Sheets APIで1行追記する。
10. Worker LambdaがLINE Push Messageで登録結果を通知する。

#### 例外フロー
1. 画像メッセージ以外を受信した場合、レシート画像を送るよう案内し、記録処理は行わない。
2. LINE署名検証に失敗した場合、不正なリクエストとして拒否し、記録処理は行わない。
3. LINE画像取得に失敗した場合、画像取得失敗としてLINEへ通知し、CloudWatch Logsへ記録する。
4. S3保存に失敗した場合、画像保存失敗としてLINEへ通知し、CloudWatch Logsへ記録する。
5. 店舗名または合計金額を読み取れない場合、スプレッドシートには登録せず、撮り直しまたは再送をLINEへ依頼する。
6. Google Sheetsへの転記に失敗した場合、読み取りはできたが転記に失敗したことをLINEへ通知し、エラー内容をCloudWatch Logsへ記録する。MVPでは自動再実行の管理画面や手動再実行機能は用意しない。
7. 同じLINEメッセージを再処理した場合、二重登録しない。

### 機能要件

#### LINE連携
- LINE Webhookで画像メッセージを受け取れること。
- LINE署名検証を行い、正規のLINEリクエストのみ処理すること。
- Webhookでは受付返信を行い、読み取り結果はPush Messageで通知すること。

#### レシート画像管理
- LINEから取得した画像をS3へ保存すること。
- S3に保存する画像は公開しないこと。
- スプレッドシートには画像の公開URLではなく、S3 URIを記録すること。
- レシート画像は90日後に自動削除すること。

#### AI Agent処理
- Agent Core Runtime上のPython Agentでレシート処理を行うこと。
- Agent実装は新規最小構成で作成し、`agentcore-sample` の既存サンプル構成に不要に依存しないこと。
- Google Cloud Vision APIのOCRテキストを事実のベースとして扱うこと。
- LLMは店舗名、日付、カテゴリ、合計金額の補正・正規化に使うこと。
- LLMはClaude Haikuを利用すること。BedrockのモデルIDまたはInference Profile IDは環境変数で変更できること。
- 日付が不明な場合は、今日の日付で補完しないこと。
- 店舗名または合計金額が不明な場合は登録しないこと。

#### スプレッドシート記録
- Google Sheets APIで対象スプレッドシートへ1行追記すること。
- レシート日付に対応する月別シートを利用すること。
- 月別シートが存在しない場合は、同じカラム構成で新規作成してから追記すること。
- 月別シート名は `yyyy-MM` 形式とする。例: `2026-05`
- レシート日付が不明な場合は、処理日が属する月のシートへ登録し、レシート日付列は空欄にすること。
- Google Sheets認証はService Accountを利用すること。
- Service Accountの認証情報はAWS Secrets Managerで管理すること。
- Google SheetsのスプレッドシートID、Service Account認証情報のSecret ARN、LINE Botの認証情報は環境変数またはSecrets Manager経由で受け取れるようにすること。
- MVPのシート列は以下とする。

| 列 | 項目 | 例 |
|---|---|---|
| A | 登録日時 | 2026-05-07T22:00:00+09:00 |
| B | レシート日付 | 2026-05-07 |
| C | 店舗名 | ローソン |
| D | カテゴリ | 食費 |
| E | 合計金額 | 1280 |
| F | 画像S3 URI | s3://receipt-images/receipts/user/message.jpg |
| G | LINEメッセージID | 1234567890 |

#### カテゴリ
- MVPのカテゴリは食費、日用品、その他の3種類に固定すること。
- 食費または日用品に判定できない場合は、その他として扱うこと。
- カテゴリは将来の月末精算や集計に使える形で保存すること。

### 非機能要件
- インフラはAWS CDKで管理すること。
- CDK環境はまずprodのみとすること。
- LINE Webhookの入口はPublic API Gatewayとすること。
- OCR、LLM、Sheets転記はSQS経由で非同期処理すること。
- 処理失敗時に再試行できるよう、SQS DLQを用意すること。
- LINEメッセージIDをシートへ保存し、WorkerやAgentの再試行が発生しても二重登録を防ぐこと。
- LINEトークン、Google認証情報などの秘匿情報はSecrets Managerで管理すること。
- LINE BotとGoogle Sheetsは実装時点で未作成でもよい。デプロイ後に必要な値を環境変数またはSecrets Managerへ設定できる構成にすること。
- 処理結果や失敗理由はCloudWatch Logsで追跡できること。
- OCR/LLM由来の値をGoogle Sheetsの数式として評価させないこと。
- LINEユーザー向け通知には内部例外、ARN、Secret名、Google Project情報などの詳細を含めないこと。
- AWS IAM権限は実行に必要なリソースへ限定すること。

## 受け入れ条件
- [ ] LINEにレシート画像を送ると受付メッセージが返る。
- [ ] レシート画像がS3へ非公開で保存される。
- [ ] 店舗名と合計金額を読み取れた場合、スプレッドシートへ登録日時、レシート日付、店舗名、カテゴリ、合計金額、画像S3 URI、LINEメッセージIDが追記される。
- [ ] レシート日付に対応する `yyyy-MM` 形式の月別シートに追記される。
- [ ] 対象月のシートが存在しない場合、ヘッダー付きで新規作成される。
- [ ] 登録完了後、LINE Push Messageで登録結果が通知される。
- [ ] 店舗名または合計金額を読み取れない場合、スプレッドシートへ登録せず、再送依頼がLINEへ通知される。
- [ ] 同じLINEメッセージが再処理されても、既存のLINEメッセージIDを検出して二重登録されない。
- [ ] S3画像が90日後に削除される設定になっている。
- [ ] Google Sheets認証情報とLINE認証情報がSecrets Managerで管理されている。
- [ ] CDKでPublic API Gateway、Lambda、SQS、DLQ、S3、Agent Core Runtime、Secrets Manager、IAM Roleが構築できる。

## 補足
将来的には、支払者、精算対象外フラグ、分担率、月次集計、精算済み管理を追加し、同棲生活の月末精算をより少ない手作業で完了できる状態を目指す。

Agent Core IdentityはMVPでは使用しない。将来、利用者本人のGoogle権限でスプレッドシートやGoogle Driveへアクセスする必要が出た場合に導入を検討する。

`agentcore-sample` は参考用サンプルであり、本機能の実装では必要な考え方だけを参照する。不要な既存構成、VPC前提、Private API Gateway前提、複数Agent前提は採用しない。
